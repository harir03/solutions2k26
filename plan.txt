FairHire — FINAL Architecture
"Inspect, Measure, Flag, Fix" — Aligned to Problem Statement
Problem Statement (Verbatim)
Computer programs now make life-changing decisions about who gets a job, a bank loan, or even medical care. However, if these programs learn from flawed or unfair historical data, they will repeat and amplify those exact same discriminatory mistakes.

Objective: Build a clear, accessible solution to thoroughly inspect data sets and software models for hidden unfairness or discrimination. Provide organizations with an easy way to measure, flag, and fix harmful bias before their systems impact real people.

How Our Architecture Maps to Every Word
PS Requirement	Our Capability	How
"inspect data sets"	Mode 1: Dataset Inspector	Upload CSV → find demographic gaps, biased labels, proxy features
"inspect software models"	Mode 2: Model Auditor	Plug in ANY model → probe with synthetic data → catch unfair patterns
"measure bias"	Fairness Metrics Engine	6 statistical metrics with formulas, thresholds, and pass/fail
"flag bias"	Severity System	Red/Yellow/Green flags with specific evidence for each finding
"fix harmful bias"	Mode 3: Fix & Prove	Sequential encoding + proxy removal → before/after proof
"easy way"	No-Code Web UI	Upload CSV, click buttons, read visual dashboard. No Python needed
"before systems impact real people"	Quarantine Gate	Biased results are HELD — never released without human review

Mode 1: INSPECT — "Find the Bias in Your Data"
What it answers: "Is our hiring data fair, or has history baked discrimination into it?"

Input
User uploads a CSV with hiring data. Example columns:

name, gender, age, college, city, skills, experience_years, hired

Six Fairness Metrics
#	Metric	Formula	Pass Threshold	What It Catches
1	Demographic Parity Ratio	(selection rate of worst group) ÷ (selection rate of best group)	≥ 0.8	Unequal selection rates
2	Disparate Impact	Same as DPR — legal name (US 4/5 rule)	≥ 0.8	Legal compliance check
3	Equal Opportunity Difference	TPR(group A) − TPR(group B)	< 0.1	Among qualified people, are chances equal?
4	Predictive Parity	Precision(A) − Precision(B)	< 0.1	Is the model equally accurate for all groups?
5	Proxy Leak Score	Max correlation of any feature with any protected attribute	< 0.6	Hidden info leakage
6	Representation Index	Min group % ÷ Expected group %	≥ 0.5	Dataset completeness
Output
Visual dashboard with 6 metric cards (Red/Yellow/Green)
Specific evidence for each flag ("hobbies column correlates 0.87 with gender")
Downloadable inspection report
Mode 2: AUDIT — "Prove Your Model Is Biased" ⭐ HERO FEATURE
What it answers: "Even if the data looks okay, is the MODEL making unfair decisions?"

Two Sub-Modes
2A: Built-in Scorer (Gemma 2 local) We provide a resume scorer powered by Gemma 2. User uploads resumes → we score them → we audit our own scorer for bias. Proves the concept.

2B: External Model (Plug any API) User provides their company's model endpoint. We send synthetic resumes to THEIR system and measure bias in THEIR model. This is the enterprise value.

Mode 3: FIX — "Remove the Bias and Prove It Worked"
What it answers: "Okay, bias exists. Now fix it and show me proof."

Three Fix Strategies (Applied Independently or Together)
Fix 1: Sequential Encoding (Our Innovation)

Every sensitive value → unique sequential code
Same value appearing twice → different codes
Original:
  Record 1: Name=Priya, College=IIT Bombay
  Record 2: Name=Rahul, College=MIT Pune  
  Record 3: Name=Fatima, College=IIT Bombay
Encoded:
  Record 1: Name=CAND_1, College=CL_1
  Record 2: Name=CAND_2, College=CL_2
  Record 3: Name=CAND_3, College=CL_3
                                    ↑
              IIT Bombay got CL_1 AND CL_3 — different codes!
              Model cannot learn "CL_1 = prestigious"
Fix 2: Proxy Removal

Features flagged in Mode 1 with correlation > 0.7
→ Removed from the dataset entirely
→ "hobbies" column dropped (was 0.87 correlated with gender)
Fix 3: Field Removal

Protected attributes removed entirely:
→ Gender: REMOVED
→ Age: converted to band (20s/30s/40s)
→ Photo: REMOVED
→ Address: REMOVED (zip code = income proxy)
